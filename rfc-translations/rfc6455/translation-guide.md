# RFC 6455 翻訳ガイド

## 翻訳品質目標

- **目標スコア**: 0.8 以上 (xCOMET評価)
- **最大反復回数**: 10回/セクション
- **10回超過時**: `needs_review: true` でスキップ、後で人間レビュー

## 翻訳スタイル規則

### 文体

- **常体（である調）** を使用
- 技術文書として客観的な表現を維持

### 技術用語の扱い

1. **カタカナ表記**: 一般的に定着している用語
   - WebSocket, プロトコル, ハンドシェイク, フレーミング
2. **英語のまま**: 固有名詞・略語
   - TCP, HTTP, URI, API, XMLHttpRequest
3. **日本語訳**: 概念を説明する一般用語
   - two-way communication → 双方向通信
   - security model → セキュリティモデル

### 改善パターン（スコア向上のため）

#### パターン1: 中黒（・）の削減

```
❌ オープニング・ハンドシェイク
✓ オープニングハンドシェイク
```

#### パターン2: 技術用語の統一

```
❌ 長いポーリング
✓ ロングポーリング
```

#### パターン3: 自然な日本語への言い換え

```
❌ オプトインしている
✓ 許可した / 受け入れた
```

#### パターン4: 文構造の調整

- 原文に忠実でありながら、日本語として自然な語順に
- 長い修飾節は前に出す

#### パターン5: 指示語の明確化

```
❌ プロトコルは...このプロトコルは...
✓ プロトコルは...本プロトコルは...
```

## 反復改善プロセス

1. **初回翻訳**: DeepL API (formality: more)
2. **評価**: xCOMET でスコア取得
3. **スコア < 0.8 の場合**:
   - エラー箇所を特定
   - 上記パターンを適用して改善
   - 再評価
4. **10回超過**: needs_review フラグを立ててスキップ

## 用語集参照

翻訳時は必ず `glossary.json` を参照し、用語の一貫性を保つ。
新しい用語を発見した場合は glossary.json に追加する。
